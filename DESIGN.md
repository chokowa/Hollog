# Hollog Design Rules

Hollog は、公開SNSではなく「自分だけのタイムライン / メディア保管庫」として使う Android 優先のローカルアプリです。デザイン判断では、見栄えの派手さよりも、毎日さっと保存し、あとで安心して探せることを優先します。

## Product Feel

- 静かで軽い。SNS風の慣れた操作感は活かすが、通知・拡散・プロフィール主役の雰囲気には寄せない。
- 主役は投稿、画像、URLプレビュー、タグ。UIの装飾はそれらを邪魔しない。
- ローカル保存の安心感を大切にする。危険な操作、消える操作、外部へ出る操作は一段わかりやすくする。
- Android 実機で片手操作しやすいことを優先する。PC Webだけで美しく見える調整は優先度を下げる。

## Platform Priority

1. Android / Capacitor 実機
2. Android WebView に近いブラウザ確認
3. PC Web / PWA

PC Web / PWA 固有の見た目や導線は、Android体験を壊さない場合だけ触ります。新しいナビゲーション、モーダル、戻る操作を追加する時は、`ROADMAP.md` の履歴制御メモを先に確認します。

## Layout

- 基本幅はスマホアプリ前提。既存の `max-w-md` / 中央寄せのモバイルシェルに合わせる。
- HOME はタイムラインが主役。ヘッダー、タグ、下部ナビ、FAB は投稿一覧を圧迫しすぎない。
- カード内にカードを重ねない。投稿カード、モーダル、繰り返し項目だけをカードとして扱う。
- 固定UIを追加する時は、下部ナビ、FAB、Android安全領域、キーボード表示と重ならないか確認する。
- 画像やメディアは安定した比率・最小高さを持たせ、読み込み前後で大きくレイアウトが跳ねないようにする。

## Color

色は `src/app/globals.css` の CSS 変数を起点にします。

- 背景: `--background`, `--secondary`
- 面: `--card`, `--muted`, `--accent`
- 文字: `--foreground`, `--muted-foreground`
- 境界: `--border`
- 主要操作: `--primary`
- 危険操作: `--destructive`
- カレンダー種別: `--calendar-post-accent`, `--calendar-clip-accent`

新しい色を直接追加する前に、既存変数で表現できないか確認します。どうしても必要な色は、ライト/ダーク両方の変数を追加します。

## Typography

- 本文・ラベルは小さくても読めることを優先する。
- コンパクトな操作面では `text-xs` から `text-sm` を基本にする。
- タイムラインや設定画面の見出しは控えめにする。大きなヒーロー見出しは使わない。
- 文字がボタンやタグからはみ出す可能性がある場合は、折り返し、短縮、最小幅、最大幅を明示する。

## Components

- ボタンは既存の `src/components/ui/app-button.tsx`、または既存画面のボタンパターンに合わせる。
- アイコン付き操作は `lucide-react` を使う。
- 主要操作は右下FAB、下部ナビ、画面内の明確なボタンに寄せる。
- アイコンだけのボタンには `title` または `aria-label` を付ける。
- 長押し、スワイプ、外側タップで閉じるUIは、誤発火しないように短い手触り確認を必ずする。
- 新しい設定は増やしすぎない。表示密度、危険操作、保存方針のように利用者の好みが分かれるものだけ候補にする。

## Motion

- モーションは短く、状態変化を理解しやすくするために使う。
- 既存の `140ms` から `260ms` 程度の範囲を目安にする。
- 投稿、画像、詳細、モーダルの出入りでは、急なジャンプよりも軽いフェード/スライドを優先する。
- ローディングや更新中は操作できない理由が伝わるように、無効化・スピナー・トーストのどれかで状態を出す。

## Interaction Rules

- Android 戻るボタン、画面内戻るボタン、モーダル終了は同じ結果になるようにする。
- HOME配下の画面遷移やモーダル終了に `window.history.back()` を使わない。
- 投稿作成、編集、インポート、削除、復元は「失敗した時にユーザーのデータが失われない」ことを優先する。
- 外部アプリへ出る、共有する、保存先を変える操作は、ユーザーが結果を予想できるラベルにする。
- タグは検索・整理・メディア閲覧の入口でもある。タップ、長押し、メニューの意味を画面ごとに変えすぎない。

## Accessibility

- タップ対象は原則 `44px` 以上にする。小さなアイコンも実際のボタン領域は広く取る。
- 文字色は `--foreground` / `--muted-foreground` を基本にし、薄すぎる装飾色を本文に使わない。
- 操作可能な要素は、押せる見た目、無効状態、選択状態が分かるようにする。
- 入力欄、検索、モーダル、メニューはキーボード表示時の重なりを確認する。

## Implementation Checklist

デザインやUIを触る時は、実装前に次を確認します。

- これは Android 実機で必要な改善か。
- HOMEタイムラインの主役性を邪魔しないか。
- 固定UI、キーボード、画像ビューア、下部ナビと重ならないか。
- ライト/ダークの両方で破綻しないか。
- タップ、長押し、戻る、外側タップ、スクロールの挙動が自然か。
- 既存の CSS 変数・コンポーネント・lucide アイコンで足りるか。
- 新しいルールが生まれた場合、この `DESIGN.md` に追記したか。

## Verification

デザイン変更後の基本確認:

```bash
npm run design:check
```

このコマンドは lint、アプリ状態シミュレーション、HOME / カレンダーのUI監査をまとめて実行します。見た目を大きく変えた時は、これに加えて Android 実機または WebView に近い環境で手動確認します。

手動確認の優先シナリオ:

- 投稿作成、編集、削除、復元
- 画像投稿、画像ビューア、画像保存
- タグタップ、タグ長押し、タグ管理
- 検索、カレンダー、設定
- 共有シートからURL/画像を受け取る流れ
- Android戻るボタン、画面内戻る、モーダル外側タップ
